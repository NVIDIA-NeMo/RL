## 4. Known upstream issues

# MANUAL: hand-tracked blockers (mirrors §"🚫 Blockers pending" in bring-up-status.md)

1. 🔴 **vLLM-DSV4 cannot load `-Base` FP8-block-quant experts.** Symptom: `fused_moe/layer.py:_load_w13 RuntimeError: size of tensor a (2048) must match tensor b (16) at dim 0`. Critical because RL policy initializes from Base, not Flash. Needs an upstream fix on PR #40760 or a follow-up PR. → tracked as `🔴 not filed yet` in §2a.
2. 🟡 **Ninja not in PATH for vLLM workers** (minor). deep_gemm JIT compiles kernels via `ninja`. Workaround: prepend `$VENV/bin` to PATH in the srun wrapper. Long-term fix in NeMo-RL worker entrypoint or base container apt layer.
3. 🟡 **DSV4 vLLM `load_format=dummy` divides by zero on FP8 block-quant requant** (`fp8_utils.py:1003`). Workaround in our recipe: `load_format: auto` (real-weight load, ~149 GB once per worker boot; first refit replaces tensors). Upstream fix landed Apr 25 on PR #40760: commit `6d244bdb4` "Support dummy loading" calls `self.finalize_weights()` lazily in the MoE forward. Our wheel `g306b63f67` predates this. Drop the override after the next `vllm/vllm-openai:deepseekv4-cu129` rebuild + re-extract + sqsh rebake.
4. 🟢 **`get_mn_major_tma_aligned_tensor` not re-exported** from `deep_gemm` top-level at `7f2a703e` (only in C++ `layout.hpp`). Soft — vLLM uses `getattr(..., None)` so import works around it. Flag if any DSV4 codepath actually calls it.
